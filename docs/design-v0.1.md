## 0. 这是什么

一个 AI PM 给一个 agent 设计了一个改动（feature）。它可以落在 agent 的任何一层——浅到一条 memory 规则、一个 skill、一段 prompt、一个工具、一个配置、一条策略，深到一段代码、乃至 agent loop 本身。改完之后，PM 想知道一件事：这个改动到底让 agent 变好了，没变，还是变差了。

不能直接问 agent，得让它在像样的情境里跑一遍看表现。这个「测试 → 跑 → 打分 → 决定改什么 → 再测」的反复过程，就是 AI PM Loop 要做成的东西。

- **形态**：开源 CLI 工具。
- **受众**：能用命令行的 AI PM / agent builder。

---

## 1. 设计原则

**可用优先。** 一切设计服从一个标准：一个真实的 AI PM 能不能真的用起来——上手成本低、不被复杂度淹没。

"极简"不单列为原则——它是个悖论：直接奔着"文件最少、命令最少"去优化，会把维持可用性的东西一起砍掉。小而干净是范围和设计做对之后的结果，不是目标；可用和"小"冲突时，可用赢。

可用优先最硬的落点是**预置优先**：不让 PM 为了跑第一个实验先造基建——接入有通用档和预置档，考题能合成，评分器带预置。

---

## 2. 抽象结构：五个元素

**① 目标描述** — PM 写的一段自然语言，说清这个 feature 想达成什么。它是根：③ 从它推导，⑤ 判断时拿它当参照。

**② 待测系统** — 被测的东西，= agent × 外部环境。一个「版本」= 两者一起冻结的快照。

**③ 评测设置** — 怎么衡量。从 ① 推导，是 ① 的可测代理，会迭代。含考题/情境 + 评分标准。

**④ 动作** — 跑 / 打分 / 比。

**⑤ PM 判断** — 读「比」的结果，拿 ① 做参照，决定下一步。不自动。

---

## 3. 怎么协同

```
① 目标描述（一段自然语言）◄───────────────────────┐
      │ 推导                                       │ ⑤ 判断时的参照
      ▼                                            │
③ 考题/情境  +  评分标准（① 的可测代理，会迭代）    │
      │         ▲                                  │
      │         └─ 代理失准 → ⑤ 决定回来改           │
② 待测版本 = agent × 外部环境                       │
      │  feature 落某一层 · 接入层负责"怎么跑"        │
      │         ▲                                  │
      │         └─ 重设计 feature → ⑤ 决定回来改      │
      ▼                                            │
④ 跑 ─► 日志 ─► 打分 ─► 分数 ─► 比 ─► 对比结果       │
   模拟/重放/固定输入  (可多次压噪声) (vs 对照组)     │
                                       │            │
                                       ▼            │
⑤ PM 判断 ──────────────────────────────────────────┘
   保留 / 回滚 / 重设计 feature / 改评测设置 / 加跑
```

这里有两段不同速度的过程：

- **快的一段（机械）**：② 和 ③ 准备好后，跑 → 打分 → 比。可重复、可自动。
- **慢的一段（判断）**：⑤ PM 读结果做决定。它的决定可以打回 ②（改 feature）、打回 ③（改评测设置），甚至打回 ①（目标本身想错了）。

## 4. 各元素第一层细节

### ① 目标描述

PM 写一段自然语言，描述这个 feature 想达成什么。就这么简单——不是结构化模板。

> 例："我设计一个 memory 机制，希望 agent 的个人化能力提上来——能记住并用上用户的历史偏好。"

它早先在讨论里也被叫作"预期效果""产品画面感""目标工程"，指同一个东西。

作用：③ 的考题和评分标准从它推导；PM 判断时拿它核对——分数高了，是不是真的等于这个目标达成了（防止 agent 学会刷分）。

### ② 待测系统

= agent × 外部环境。

- **agent**：model + harness + prompt + skill + memory，即 agent loop 运行时的执行与调用逻辑。feature 通常改这里某一层。
- **外部环境**：平台（mac/win）、沙箱、外部数据源。
- 边界：prompt/skill/memory 归 agent 内侧；平台/沙箱/外部数据源归外部环境。也可用"内环境/外环境"称呼，同一刀。
- 不同 agent 类型（Codex / Claude Code / Dify workflow 等）接入方式不同，见 §5。

### ③ 评测设置

**考题/情境**——三种来源：

- 手写。
- 从 ① 合成：① 既然写下来了，工具可据它草拟候选情境，PM 来 curate。
- 从真实日志抽取：把跑过的真实会话回灌成可复跑的考题。

**评分标准**——维度 + 权重。出厂带几套预置 rubric（通用任务完成度、对话质量、指令遵循 等），PM 选一套改，不从零写。

### ④ 动作

**跑**——产出行为日志，三种模式：

- 模拟：LLM 扮用户跟 ② 交互。
- 重放：把真实日志的轮次放给 ② 重跑。两个口子——接某个数据源，或直接放一个原始数据文件。
- 固定输入：写死的输入脚本。

**打分**——日志按 ③ 的评分标准打分。可对同一批日志重复打多次，用来压测量噪声。

**比**——把一次结果跟一个**对照组**比。最常见的对照组是基线（没装 feature 的那版）。也可以是别的版本——比如让 agent 自己进化出来的版本（即对比 PM 协同进化 vs agent 自主进化）。比的时候看三件事：差异有没有超过噪声地板、有没有哪个维度退化、差异该归因到 feature 本身还是评测设置的干扰。

> 注："比"就是"跟对照组比"这个动作。上面 PM-vs-自主进化只是其中一种比法，不是"比"的通用定义。

### ⑤ PM 判断

读"比"的结果，拿 ① 做参照，决策：保留 feature / 回滚 / 重新设计 feature / 改评测设置（① 的代理失准了）/ 加跑压噪声。这一步是 PM 的，不自动。

## 5. 待测系统接入设计

接入层干两件事，分开看：

### 驱动对话（发一轮、收一轮）

可以通用，分三档：

- **通用 API，零代码**：② 若能通过标准协议触达——Chat Completions（最通用，作默认）、Responses API（带服务端状态的 agent）、Managed Agents——PM 给 endpoint + 协议 + auth 即可。
- **预置接入**：不讲标准 API 的常见 agent——Claude Code（CLI）、Codex、Dify workflow，以及本地 runtime 型 agent——工具自带 adapter。
- **自定义接入**：兜底，实现两个函数（驱动一轮 / 布置一个版本）。

### 布置版本（装上 feature、配好 agent + 外部环境、可选快照重置）

agent-specific，做不到完全通用。但常见用法可以很轻：PM 准备好版本 A、版本 B 两套状态，工具分别指过去即可。沙箱快照/重置那种重型布置只在做严谨对照实验时才需要，交给预置 adapter，不进内核。

一句话：驱动通用化；布置默认轻（两套状态），重型可选。
