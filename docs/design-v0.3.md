## 0. 这是什么

一个 AI PM 给 agent 设计了一个改动。改动可以落在任何一层——一条 memory 规则、一个 skill、一段 prompt、一段策略、一段代码。

改完，他想知道一件事：**这个改动让 agent 变好了，没变，还是变差了？**

不能直接问 agent，得让它在像样的情境里跑一遍、看表现。这个「定方向 → 设计实验 → 跑 → 打分 → 比 → 判定」的反复过程，就是这个工具要做的事。它是一个开源 CLI 工具，给会用命令行的 AI PM 和 agent builder 用。

**一条范围线**：工具做到「判定」为止。它只读 agent，不写 agent。判定说「这个改动值得用」之后，真把它用到你的生产 agent 上——那是你在工具外做的事，本版不设计。

---

## 1. 两条原则

**可用优先。** 所有设计只服从一个标准：一个真实的 AI PM 能不能真的用起来。「极简」不单列为原则。文件少、命令少是设计做对之后的副产品；直接拿它当目标去优化，反而会把维持可用性的东西也砍掉。

**预置优先。** 接入方式、评分器、模拟器都出厂带预置，PM 不用为了跑第一个实验先造一堆基建。但预置只是起点。评测设置最终要从目标推导，预置只是让你不从零开始。

## 2. 抽象：三层，两个角色

整件事是三层套在一起的：

```
总目标    你想把这个 agent 变成什么样
   │
   │  拆成一串实验
   ▼
实验      一个假设 + 一组版本 + 一套评测设置
   │
   │  交给 coding agent 跑
   ▼
版本      几个「agent × 环境」的快照，摆一起跑、打分、比
   │
   ▼
对比结果 → 判定 → 回到上面某一层去改
```

- **总目标**——你要把这个 agent 变成什么样，写在 `goal.md` 里（内部结构见 §2.1）。它会一直被修正：每轮实验有一部分作用，就是检验总目标本身对不对。
- **实验**——一个个实验，每个朝总目标走一步。一个实验 = 一个假设 + 一组版本 + 一套评测设置。
- **版本**——一个实验里，摆出来对比的那几个 agent 快照。

干这件事的有两个角色：

- **PM**，也就是人。管总目标、设计实验、写 program、做判断。
- **coding agent**，PM 手上用的那个——Cursor、Codex、Claude Code 都行，是个槽位。它是真正动手的：搭骨架、抓环境、起草版本、跑、打分、比、按规则判。

中间那个 **program**，是 PM 交给 coding agent 的东西，一个实验一份。里面写：这次的假设、留/丢规则、喊人规则，还有实验的几项声明（下面会讲）。

### 2.1 goal.md 的三层

`goal.md` 里写的总目标，自己再分三层。分开写，是有意的：

- **元目标（meta-goal）**——定性的，你真正想要的。不直接拿去优化。
- **操作目标（operational goal）**——可验证的，是 agent 实际在优化的东西。它是元目标的一个代理（proxy）。
- **评测设置（eval suite）**——操作目标怎么量。`goal.md` 这层定的是目标级的口径；每个实验再据此搭自己那套测试集／评分器（见 §3.3），实验级的 rubric 从这里推导。

为什么要拆：操作目标是个代理，代理会被钻空子（Goodhart）。把元目标单独留着、跟操作目标摆在一起，是对 Goodhart 的明置防线——也正是外层循环要干的活：盯住「指标涨了」和「我真正要的东西变好了」之间那道缝。

总目标内部就是这三层。再往里，每一层具体怎么写、怎么 engineer，是 Kun 的 Goal Engineering 方法论，仍待补。

## 3. 一次实验由什么组成

一次实验要备齐三摊东西：被测的系统、怎么连上它、拿什么评测。

### 3.1 被测系统

被测的，是一个 agent 的环境。

一个实验里，你摆几个版本来对比——`V1`、`V2`、`V3`，每个起一个 sandbox。其中一个不动，当参照。起几个、各装什么，由这次的假设定。

每个版本是一份**环境快照**。环境分五层，按「能不能快照」来分：

1. **可版本化文件**——记忆、skill、prompt、config、harness 代码。
2. **标识符**——model、平台。记一个字符串就行。
3. **大块状态**——state.db、runtime 镜像。
4. **依赖**——venv、lockfile。
5. **外部、会漂的**——平台、sandbox、机器状态、外部数据源、背后的 LLM API、web、MCP。这层快照不了，只能记下来、打个时间戳。

抓环境是个过程，得一层层来。能抓多少抓多少，抓不到的就明写「这层没抓全」。到底能取多全，完全看那个 agent。

第一个实验抓到的那份基线，会冻成一个顶层的 `origin`——永不更新的初始态。

还有两件要在这里声明的事：

- **快照节奏**：被测 agent 一轮里不改自己，开头抓一次就够；要是它边跑边自己长 skill，就得每轮抓一次。
- **有环境 / 无环境**：上面说的都是「有环境」。如果你只有一个 API、没有环境可抓，那就是「无环境」——这时一个版本就塌成一个请求配置（model、system prompt、参数）。无环境这档轻，适合单模型、单 agent 评测。

### 3.2 接入

工具怎么够到被测 agent。四种，出厂都带：

1. **进程内库**——agent 是个能 import 的库，比如 <experiment>。
2. **外部命令行**——agent 是个独立程序，用子进程加 JSON 通信，比如 openclaw。
3. **HTTP，无状态**——Chat Completions 那种，每轮把完整对话历史发过去。
4. **HTTP，服务端有状态**——Responses API、托管 agent。

接入方式还决定一件事：环境能快照得多完整。本地那两种，能完整快照。托管那两种，够不着文件，只能记一个配置引用。

### 3.3 评测

评测要三样东西：测试集、对话模式、评分器。

**测试集**，是一组测试case。一个测试case = 一段起始输入（初始上下文 + 第一个任务），可选再带一条完成标准。测试集从哪来：手写、从 `goal.md` 合成、或从真实日志里抽。它没有通用预置。

**对话模式**，决定一段对话怎么生出来。三种：

- **模拟**——一个模拟器扮用户。模拟器 = 人设 + 背景知识 + 追问策略。它从起始输入起，一轮轮现生对话。模拟这档特殊：测试集里只有起始输入，而且要单独配一个模拟器。
- **回放**——把录好的真实对话喂进去。这时一个测试case 就是一整段完整对话。
- **固定**——按写死的脚本跑。一个测试case 就是一整段脚本。

**评分器**，把对话变成分数。它先有个类型：

- **规则脚本**——确定性代码，比如正则、精确匹配、查有没有调工具。没噪声，不花钱。
- **LLM Judge**——一个 LLM 照 rubric 打分。有噪声，花钱。「用哪个 LLM」这个设定，只有这一档有。
- **组合评分**——客观项交给规则脚本，主观项交给 LLM Judge。

评分器里的 rubric（维度加权重），从 `goal.md` 推导——可以选个预置改，也可以照 goal 草拟一版。rubric 是整套东西里改得最勤、也最容易出事的：一版坏 rubric 会悄悄放大噪声。

## 4. 三个动作

备齐上面那些，就是三个动作：

- **跑**——让每个版本过一遍测试集，产出对话。
- **打分**——拿评分器给对话打分。跑和打分是分开的：打分便宜、能重打（换 rubric、换评分模型再打一遍）。每份分数都标清楚是哪版 rubric、哪个模型、对应哪次跑。
- **比**——把一组版本放一起比。看四样：差异有多大；差异有没有超过噪声（前提是量了噪声）；哪个维度退化了；这点差异该归因到 feature，还是评测设置的干扰。

关于噪声：压噪声（多跑几次 trial、多打几遍分）是**可选**的，按这次实验的需要来。预期效果很大、或只想快速看一眼，就跳过。效果小、又要据此拍板，才值得花功夫去压。

## 5. 完整流程

主角是 coding agent，PM 只在要做判断的地方出手。

**〇 一次性接上。** 配好接入（四种选一），写下 `goal.md`（总目标）。这步只做一次。

**一 开一个实验。** 按「实验的三块」来。每块都是先声明类型、再建文件——声明它是什么类型，本身就决定了它要哪些文件：

- 建被测系统：声明有/无环境、快照节奏，然后抓基线环境，起 `V1`、`V2`、`V3` 几个 sandbox。
- 建评测：声明对话模式、评分器类型，然后备好测试集和评分器；模拟模式还要再备一个模拟器。
- PM 写 program：假设（扣着 `goal.md`）、留/丢规则、喊人规则。

**二 跑。** coding agent 读 program，先报一个成本和时间的预估（PM 可以设上限），然后跑、打分、比，出一份对比报告。比完它拿喊人规则对一下，撞上就停下、把结果交给 PM。

**三 判定。** 谁来判，看用的是哪个模式（见 §6）。判的人读对比报告，定下一步。下一步落在三层之一：

- **版本层**：接受一个改动（记下「值得用」）、丢掉它、或者重新设计一个再试。
- **实验层**：加跑几次压压噪声，或者改测试集、改评分器再重跑。
- **总目标层**：改 `goal.md`。

这里没有「赢家」一说。很多实验的结论就是「这改动没过噪声」「有得有失」——那也是有效结论。不管走哪条，整个实验都原样留着：program、版本、评测、results、判定，合起来是一份能复现的记录。

（没有 step 四。「应用到生产 agent」本版不做，见 §0 和 §8。）

## 6. 两个模式

「人参与多少」做成两个能直接选的预设：

- **人评**——coding agent 只管跑。设计、判定、定下一步，都 PM 来。一次一个实验，PM 每轮都在。
- **agent 自迭代**——coding agent 把整个循环包了：照 `goal.md` 设计一个改动、跑、按 program 判、再设计下一个，连着转很多轮。PM 只管 `goal.md` 和 `program`。

这两个是两端。喊人规则在中间做微调——比如自迭代模式里，它可以说「碰到某维度退化、或差异贴着噪声线，也停下来交 PM」。

两个模式都只在实验空间里转，不碰你真实/生产的 agent。

## 7. 文件结构（待细化）

整条流程走下来，文件结构大致是这样：

```
ai-pm-loop/
├── goal.md                总目标
├── origin/                冻结的初始环境快照
├── agent/connect          接入配置（四种选一）
├── experiments/
│   ├── index.md           实验链索引
│   └── 001-xxx/
│       ├── program.md     假设、声明、留丢/喊人规则
│       ├── V1/ V2/ V3/    几个版本的 sandbox
│       ├── 测试集/  评分器/  模拟器/   （模拟器只在模拟模式有）
│       └── results/       对话、分数、对比报告、判定
└── （工具本体和命令）
```

文件名都是占位。每个文件里具体装什么，是下一步要定的。
