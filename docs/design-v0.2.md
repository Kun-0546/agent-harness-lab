## 0. 这是什么

一个 AI PM 给一个 agent 设计了一个改动（feature）——它可以落在 agent 的任何一层，浅到一条 memory 规则、一个 skill、一段 prompt、一个配置，深到一段代码、乃至 agent loop 本身。改完，PM 想知道一件事：这个改动到底让 agent 变好了，没变，还是变差了。

不能直接问 agent，得让它在像样的情境里跑一遍看表现。这个「定方向 → 设计实验 → 跑 → 打分 → 比 → 决定」的反复过程，就是 AI PM Loop 要做成的开源 CLI 工具。

- 形态：开源 CLI 工具。
- 受众：能用命令行的 AI PM / agent builder。

---

## 1. 设计原则

**可用优先。** 一切设计服从一个标准：一个真实的 AI PM 能不能真的用起来——上手成本低、不被复杂度淹没。

"极简"不单列为原则——直接奔"文件最少、命令最少"去优化，会把维持可用性的东西一起砍掉。小而干净是范围和设计做对之后的结果。

可用优先的落点是**预置优先**：接入有预置、考题能合成、评分器带预置，PM 不用为跑第一个实验先造基建。但预置只是让你不从零开始——评测设置最终还是要从目标推导出来。

## 2. 抽象结构：三层

整件事是三层套着的。上面两层是 PM 的，最下面一层交给一个 agent。

```
总目标   你想把这个 agent 变成什么样
  │      PM 写，而且一直在修正——实验的发现会反过来改它
  ▼ 拆成一串
实验     一个小问题（假设）+ 一组变体 + 一套评测设置
  │      PM 设计每一个，每个朝总目标走一步
  ▼ 交给 agent
变体     一组「agent × 环境」的快照，摆在一起
         agent 跑 → 打分 → 比，出一份对比结果
  │
  ▼
对比结果 → PM 读 → 往上回改：换变体 / 改这个实验 / 改总目标
```

- **总目标层** — 你要把这个 agent 变成什么样。PM 写，并且持续修正。
- **实验层** — 一个个实验，每个问一个小问题、朝总目标走一步。一个实验 = 一个假设 + 一组变体 + 一套评测设置。
- **变体层** — 一个实验里摆出来对比的那组「agent × 环境」快照。agent 把它们跑出来、打分、比。

三层各有一个循环，速度不同：

- 变体层（快、机械）：跑一组变体 → 打分 → 比。agent 干。
- 实验层（慢、判断）：PM 看完一个实验，设计下一个。
- 总目标层（更慢、但持续）：实验的发现反过来修正总目标。

> 和 v0.1 的关系：v0.1 的五元素并进了三层——①目标 = 总目标层；③评测设置 + 设计实验 = 实验层；②待测系统 + ④跑/打分/比 = 变体层；⑤判断 = PM 在上两层上做的动作。

## 3. 两个角色 + program

这件事只有两个角色：

- **PM（人）** — 管总目标、设计实验。把一个实验写成一份 **program** 交给 agent。
- **agent（一个 coding agent）** — PM 手上用的那个，Claude Code / Codex / 别的都行（是个槽位，不写死）。读 program，把这个实验跑完：跑变体、打分、比、按规则定留/丢、该停下来就交还 PM。

**program** 是 PM 交给 agent 的东西，一个实验一份。它写清三样：这个实验怎么跑；留/丢按什么规则判；以及——**什么情况下 agent 别自己拍板、停下来交还给 PM**。最后这条是「人参与多少」的旋钮：调到底是全自动，往回调就是每步都让 PM 看。

## 4. 三层细节

### 总目标层

PM 写下「想把这个 agent 变成什么样」。它写下来之后还会一直被改——每轮实验除了测一个变体，还有一个作用：检验总目标本身对不对。这一层就是目标工程。

> 这一层内部怎么写、怎么修正一个总目标，是 Kun 的目标工程方法论的地盘。本版先把这层的位置留好，内部细节待补。

### 实验层

一个实验 = 一个假设 + 一组变体 + 一套评测设置。

- **假设** — 这个实验想回答的一个小问题，一两句话。
- **一组变体** — 见变体层。含一个基线（没改的那版）。
- **评测设置** — 考题/情境 + 评分标准。
  - 考题：手写 / 从总目标合成（工具草拟、PM curate）/ 从真实日志抽取。
  - 评分标准：出厂带几套预置，PM 选一套改。它是整套里变得最勤、也最危险的东西——一版坏标准会悄悄放大测量噪声。所以它要能记版本、能回退；一版新标准用之前，先单独量它自己的噪声。

### 变体层

一个变体 = 「agent × 环境」冻下来的一份完整快照。

- agent（内环境）：model + harness + prompt + skill + memory。
- 外部环境：平台、沙箱、外部数据源。
- feature 就是某个变体跟基线之间的差。
- 一个变体的快照必须存下它那套真实环境文件——否则过段时间认不出哪个分数是哪个版本跑的。能存多完整，取决于接入方式（见 §5）。

三个动作：

- **跑** — 产出对话日志。三种来源：模拟（LLM 扮用户）/ 重放（喂真实日志的轮次）/ 固定输入。
- **打分** — 日志按评分标准打分。同一批日志可重复打多次，压测量噪声。
- **比** — 一组变体一起比。看三件事：差异有没有超过噪声、有没有哪个维度退化、差异该归因到 feature 还是评测设置的干扰。每个分数标清：哪版评分标准、哪个评分模型、对应哪次跑。

## 5. 接入

工具怎么够到被测 agent，四种，出厂带：

1. **进程内库** — agent 是个能 import 的库（如 <experiment>）。同进程，工具对它环境的控制最强。
2. **外部命令行** — agent 是个独立程序，当子进程起、命令行 + JSON 通信（如 openclaw）。跨语言。
3. **HTTP・无状态** — Chat Completions 那种，每轮把完整对话历史发过去。
4. **HTTP・服务端有状态** — Responses API、托管 agent。对话状态在服务器，工具只攥一个会话编号。

都不沾，留一个自定义口子（实现一个小约定：送一轮 / 收一轮 / 用一套环境文件起一个实例）。

**一条要紧的耦合**：接入方式决定变体层那个「环境快照」能多完整。前两种（本地）——环境是本机文件，能完整快照、能重置；后两种（托管）——环境在别人服务器上，工具够不着，只能记一个配置引用。工具接本地 agent 时是满配，接托管 agent 时是打折版。

## 6. 闭环、实验链、成本

v0.1 没有这三样，补上：

- **闭环** — 一个实验跑完、某个变体赢了，得有一步把这个赢的改动并回 agent 本身（更新「当前 agent」状态），下一个实验从更新后的状态起。少了这步，agent 永远不动，工具就只是在产报告。（并回去是一直累积、还是每次从干净状态重起，两种都要支持。）
- **实验链** — 实验越积越多，要一个索引（自动生成），一行一条，看清它们怎么一步步朝总目标走。
- **成本** — 跑一次烧钱烧时间。跑之前给个预估，能设上限（几条考题、每条几轮、打几次分）。
